# Realtime Recording Subtitles

## Problem Frame
当前录音流程只能采集音频并在结束后上传，用户在录音过程中看不到系统是否已经“听懂”内容，也无法在走神、漏听时通过字幕快速跟上内容。为了给后续更强的录音能力铺路，需要把录音过程升级为一条可持续累积的实时字幕链路：录音中先看到实时原文，原文稳定后再看到翻译，并在结束录音后尽量直接复用这条链路产出的字幕结果，而不是重新从头生成。

## Requirements
- R1. 当用户开始录音后，界面应尽快开始显示实时原文转录，让用户在录音过程中持续看到已经识别出的内容。
- R2. 实时原文允许被后续上下文回改，以提升识别准确率；界面应允许用户感知“仍可能变化”的内容与“已经较稳定”的内容存在区别。
- R3. 实时翻译不应在原文刚出现时立即展示，而应在对应原文达到相对稳定后再显示，以减少双语内容频繁抖动。
- R4. 录音过程产生的原文与翻译结果应被持续保留，作为录音结束后正式字幕的主要来源，而不是仅作为临时预览。
- R5. 当用户结束录音时，系统应直接复用大部分实时字幕结果，仅对最后尚未稳定的尾部内容做收尾处理，而不是重新对整段音频做一次完整字幕生成。
- R6. 结束录音后，如果后续流程需要翻译字幕、生成视频或进入其他内容处理步骤，应优先消费这份已累计的字幕结果，避免重复等待与重复计算。
- R7. 首个阶段的重点是打通“实时字幕可见且可复用”的主链路，而不是追求所有录音后处理都在实时阶段一次完成。

## Success Criteria
- 用户在录音开始后的短时间内就能看到实时原文，而不是必须等录音结束。
- 用户在录音过程中可以通过字幕理解老师刚刚讲了什么，即使短暂走神也能快速跟上。
- 实时翻译只在原文相对稳定后展示，双语体验不会因为过度抖动而失去可读性。
- 录音结束后，大部分字幕内容直接来自实时阶段的累计结果，只对尾部未稳定内容做补全。
- 后续任何依赖字幕的流程都可以复用这份结果，不需要默认再从头生成一份字幕。

## Scope Boundaries
- 本次不要求把所有录音后处理都改造成完全实时完成。
- 本次不要求在实时阶段就保证最终字幕百分之百定稿，尾部补全和少量收尾修正是允许的。
- 本次不要求定义新的复杂编辑工作流，例如让用户手动逐句确认或修改实时字幕。
- 本次不要求在 brainstorm 阶段确定具体模型、浏览器 VAD 方案、分段协议或存储结构。

## Key Decisions
- 实时链路定位为正式产物前置，而不是纯预览能力: 录音中的字幕结果要在结束后继续复用。
- 原文允许回改: 优先提高识别准确率，而不是追求一旦显示就绝不变化。
- 翻译延后到原文稳定后再显示: 优先保证双语可读性与信任感，而不是最短延迟。
- 结束时只做尾部收尾: 目标是让“录音中已经做过的工作”直接沉淀为正式字幕，避免整段重跑。

## Dependencies / Assumptions
- 录音入口仍然是当前的实时录音流程，新增能力是在该流程上扩展，而不是另起一个独立录音产品。
- 产品已有字幕与翻译消费场景，因此复用实时结果会对后续流程产生直接价值。
- 用户对实时结果的核心诉求是“看见系统已理解到哪里”，其次才是把后处理整体提速。

## Outstanding Questions

### Deferred to Planning
- [Affects R1,R2][Technical] 如何定义“稳定”和“未稳定”的状态，并在前端上用最小心智成本表达出来。
- [Affects R4,R5][Technical] 实时链路的中间结果以什么粒度累积、更新、持久化，才能既支持回改又支持结束后直接复用。
- [Affects R3,R6][Needs research] 翻译应以句子、短段落还是时间窗口为单位触发，才能平衡延迟、成本与稳定性。
- [Affects R5][Technical] 尾部收尾处理的边界应该如何定义，避免既补不全又意外触发整段重算。
- [Affects R6][Technical] 现有依赖字幕的后续流程中，哪些可以直接消费实时累计结果，哪些还需要额外的最终化步骤。
